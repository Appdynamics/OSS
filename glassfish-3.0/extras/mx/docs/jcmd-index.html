1. Introduction

JCmd ('jcmd') is an all-Java command line interface shell. It supports the following:

  • a powerful framework for implementing commands
  • an extensible (pluggable) command set based on a developer-selectable mix of built-in and pluggable comands
  • a CLI compliant with SUSv3 and Sun Microsystems CLIP standards
  • invocation either from a command shell or in interactive mode.

Javadoc is available. If you are interested in adding or removing commands from JCmd, see the optional facilities documentation.If you want to build your own command line, see the documentation.

2. Overview

The following sections discuss some key concepts used by jcmd which should be understood to use it effectively.

2.1 JCmd command shell

JCmd may be invoked either conventionally via one's usual command shell, or in an interactive mode. Example:

Single invocation:

bash-2.05$ jcmd help
...

Interactive mode:

bash-2.05$ jcmd
jmxcmd> help
...

State is saved by jcmd in the user's home directory. State includes the jcmd environment variables as well as the command history.

Most commands have abbreviations available; refer to the help for each command. All commands have both long and short names for command options (if any).

2.2 Startup performance

Many command-line users are concerned about the startup-time penalty of using a Java-based CLI. Compared to a C-based executable, Java is relatively slow. The startup of jcmd should be comparable to starting up a java "hello world" application.

The JCmd framework further mitigates this problem in several ways.

(1) Where possible, "lazy" initialization is performed. The assumption is that only a minor portion of the facilities may be used, so no advance initialization is used, nor is the base architecture dependent on large numbers of classes.

(2) Complex, CPU-intensive initialization is avoided by avoiding the use of complex configuration (eg XML), which can consume substantial amounts of CPU time to parse.

(3) When multiple commands are to be issued interactively, the user may start the CLI once, and issue multiple commands (interactive mode). In such a case, the startup overhead becomes irrelevant.

(4) When multiple commands are to be issued programmatically, they may be grouped into a file, and the source command may be used to execute them. In this case, there is only one startup of jcmd for the entire lifecycle of the source command, which could execute an arbitrary number of commands.

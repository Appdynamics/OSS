2. Commands detail

2.1 get --display one or more attributes on the specified target(s)

get [--verbose|-v] attr[,attr]* [target]+

The 'get' command displays MBean attributes for all specified target(s). If no targets are specified, then the default target it used, as last set by the 'target' command. Attributes should be specified in a comma-separated list. In addition, the following special targets are supported:

Target Meaning
* All attributes
*r all read-only attributes
*w all writeable attributes

The get command is robust in the face of failures; failures in getting some Attributes of an MBean will not prevent the other Attributes from being accessed. Situations that can cause this include non-serializable Attributes or Attributes for cannot be deserialized due to the absence of client-side classes for them.

Examples

1. Get all attributes on all MBeans:

jmxcmd> get * *

Note that if you invoke jmxcmd from within a command shell, then you will need to quote the "*".

2. Get the 'Count' and 'Timeout' attributes on the MBean(s) referred to by the alias 'MyMBean':

jmxcmd> get Count,Timeout MyMBean

 

2.2 set --set one or more attributes on the specified target(s)

jmxcmd> set attr=value[,attr=value]* [target]+

The 'set' command sets the specified MBean attributes for all specified target(s). If no targets are specified, then the default target it used, as last set by the 'target' command. Attributes should be specified as a series of name-value pairs in a comma-separated list. In addition, the following special targets are supported:

Examples

1. Set the 'Count' and 'Timeout' attributes on the MBean(s) referred to by the alias 'MyMBean':

jmxcmd> set Timeout=10,Count=20 MyMBean

Note that the comma-separated list of name-value pairs must not include white space before or after the ','.

 

2.3 find --display MBeans matching name, pattern, or alias

find [--terse] [--wild-prop|-w=<expr=expr> | --jregex-prop|-j=<expr=expr>] [--current|-c ] [--add|-a] [--remove|-r] [target]*

find --current|-c [--terse]

Finds mbeans within the set of MBeans denoted by the target(s). Each operand to the find command may be a fully qualified ObjectName, an ObjectName pattern, or an alias. The set of these operands and ultimately what they resolve to defines the starting set upon which any further selection is done. Certain special characters may be used with the targets:

  • the '!' character may be used to reverse the set that the target selects.
  • with more than one operand, the '+', '-' and '&' characters modify the set of targets so far, from left to right. The '-' operator substracts the target from the set so far, the '+' operator adds to it (same as no qualifier), and the '&' character performs a logical and of the target and the set so far.

The MBeans found will be displayed unless the --terse option is used, in which case only a total count will be given.

The 'find' command automatically maintains the alias "last-found" with the results of its last invocation.

Option Explanation
--terse | -t display a count only of the MBeans found
--current | -c display the results of the last find (current set). This will be the same as the contents of the alias "last-found".
--add | -a add the results of this invocation to the current set
--remove | -r remove the results of this invocation from the current set
--wild-prop | -w utilize a wildcard expression for MBean Property name(s) or value(s). The only allowed wildcard is the '*' character.
--jregex-prop | -j utilize a regular expression for MBean Property name(s) or value(s) based on java.util.regex patterns
--operations | -o comma-separated list of (possibly wildcarded) expression(s) for operation names
--attributes | -a comma-separated list of (possibly wildcarded) expression(s) for Attribute names

When specifying --attributes or --operations, MBeans that match any of the operations or attributes are selected (logical OR). The same holds true for --regex and --jave-regex.

Examples

1. Display all MBeans:

jmxcmd> find

2. Display all MBean in the default domain:

jmxcmd> find :*

3. Display all MBeans whose ObjectName contains the property 'type' with value 'test'

jmxcmd> find type=test

4. Displays all MBeans whose ObjectName contains a property whose name starts with 'n' and whose value starts with 'test'

jmxcmd> find --wild-prop=n*=test* *:*

5. Displays all MBeans not denoted by the alias "MyMBeans":

jmxcmd> find !MyMBeans

6. Displays all MBeans which have the property-pair "category=config" and that are also in the domain "Test":

jmxcmd> find category=config &Test:*

7. Displays all MBeans that have an operation containing the string "Test", "test", or "TEST" or an attribute containing "Count" or "count" in the default domain that don't have an ObjectName property "category=config":

jmxcmd> find --operations=*Test*,*test*,*TEST* --attributes=*count*,*Count* :* -category=config

9. Inspects all MBeans last found with the find command

jmxcmd> inspect last-found

 

2.4 inspect --display MBeanInfo (attributes, operations, constructors, notifications)

inspect [--all|-A] [--no-description|-d] [--attributes|-a=attr[,attr]*] [--operations|-o=op[,op]* ] [--constructors|-c] [--notifications|-n=notif[,notif]*] [--include--nomatch|-i] [target]*

One or more targets may be specified. If no targets are specified, then the default target is used. All matching MBeans will be displayed with their fully-qualified names.

Option Explanation
--all | -A display all available information
--no-description | -d omit descriptions (avoids clutter if there are none)
--attributes | -a display the specified attributes (* = all attributes). A comma-separated list of attribute names may be used, and each of them may be wildcarded using the "*" character.
--operations | -o display the specified operations (* = all operations). A comma-separated list of operation names may be used, and each of them may be wildcarded using the "*" character.
--constructors | -c display all available constructors
--notifications | -n display the specified notifications (* = all notifications)
--include-nomatch | -i include all MBeans specified by the targets, even if no attributes, operations, constructors or notifications are found that match the request. A summary only of those MBeans will be displayed.
 
Examples

1. Display all information on all MBean(s) referred to by the alias 'MyMBean':

jmxcmd> inspect --all MyMBeans

2. Display all attributes available on all MBeans:

jmxcmd> inspect --attrs * all

2. Display attributes containing "count" or "Count" on all MBeans

jmxcmd> inspect --attrs *count*,*Count* *

 

 

2.5 invoke --invoke an MBean operation

opname:[arg[,arg]*] [target[ target]*]
alias-name.opname:[arg[,arg]*]

opname([arg[,arg]*) [target[ target]*]
alias-name.opname([arg[,arg]*)

In this syntax, 'opname' is an operation name, alias-name is an alias as created by the create-target-alias command, 'arg' is an argument value or a named-argument (name/value pair) and 'target' is either an alias or ObjectName. An invocation uses wither Ordered Invocation, in which the order of the parameters is explicit according to the order they are listed, or Named Invocation, in which parameter order is determined by the MBeanParamerInfo using the parameter names. Note that Named invocation is not available if an MBean does not supply parameter names. You can use the inspect command to see if parameters have names.

Named invocation also works for operations taking a Properties object; excess parameters are supplied to the operation in the Properties object.
All forms of invocation use the MBeanInfo to determine the correct match for an operation.

For an operation to be available for invocation, it must restrict its use of data types to the following:

  • char, byte, short, int, long, float, double
  • Character, Byte, Short, Integer, Long, Float, Double, Number, BigNumber
  • String
  • Object
  • Properties
  • [] all arrays of the above types


Type-casts may be used, but are rarely needed. Use a type cast to force a number to a String or int to Integer, etc. Arrays are denoted using curly braces, and may be nested. Examples of arrays:

{1,2,3}
{hello,there}
{hello,1,there,2}
{{1,2},{3,4}}

If a type-cast is applied to an array, then all elements of that array must be compatible with it. The following type-cast
forces the value to be converted to an array of String (which would match 'String []' in an operation):

(String){1,2,3}


Strings may be quoted with the double-quote character '"'. This is not required, but can be useful to force a value's type
to be a String where it could otherwise be interpreted as a number.

Target(s) of the invocation may be specified in three ways:

  • implicitly by omission (the default target will be used as set by the target command);
  • explicitly as operands ;
  • explicitly as an alias name preceeding the invoked operation.

The operation may be specified using two syntactical forms:

  • the ':' form
  • the method invocation form ()

All forms are equivalent in function.

This table lists all variants:

# Variant Comments
1 opname:[arg[,arg]*] [target]+ invokes on one or more specified targets, which may be aliases or ObjectNames
2 opname:[arg[,arg]*] invokes on the default target(s) as set by the target command.
3 opname([arg[,arg]*]) [target]+ same as variant 1, but with () instead of ':'
4 opname(arg[,arg]*]) same as variant 2, but with () instead of ':'
5 alias-name.opname:[arg[,arg]*] invokes on the target(s) defined by the alias 'alias-name'.
6 alias-name.opname([arg[,arg]*]) Same as variant 5, but with () instead of ':'


Option Explanation
Examples

1. Invoke the start() operation an an object named "mydomain:name=controller":

jmxcmd> start() mydomain:name=controller

jmxcmd> start: mydomain:name=controller

2. Invoke the add( int x, int y) operation on an mbean with the alias 'math'

jmxcmd> add:3,4 math

jmxcmd> math.add(3,4)

2.12 connect--manage connections to MBeanServers

connect [--host|-h host-or-ip] [--port|-p port-number]
[--user|-u username] [--password-file|-f path] [--prompt|-P]
[--truststore-file|-t path] [--protocol|-r protocol-id] [--options|-o name=value[,name=value]*] [connection-name]

list-connections
close-connection [connection-name]

There are several variants here, described in the following table:

Name Explanation
connect Connect to the default server. The default server is the one last accessed.
connect connection-name Connect to a server, using the parameters with the name connection-name. The connection-name must have been previously created.
connect [<options>] Make a new connection using the specified parameters and name it.
list-connections Same as "connect -l" above.
close-connection [connection-name] Close the named connection, or close the default one if no name is supplied

When jmxcmd makes a connection to an MBeanServer, it relies on pluggable modules implementing the JMXConnectorProvider interface (com.sun.cli.jmxcmd.spi.JMXConnectorProvider). Each connector has its own parameters, which may be a subset, superset, or alternative set to the ones shown in the above syntax. However, for user convenience when working with more than one connector, certain options are made available as common names. These include host, port, user, and password-file. The protocol option is always used to specify the protocol, but defaults to jmxmp, the TCP/IP based JSR 160 protocol for accessing remote MBeanServers.

The following options apply to the connect command:

Option Explanation
--user|-u username The username to authenticate with.
--password-file|-f path The password file containing username/password pairs.
[--prompt|-P] Prompts for passwords as needed. No prompting will occur if --password-file is used and the password is found.
--host|-h host-or-ip The hostname or IP addresss to pass to the Provider used in connecting to the server.
--port|-p port-number The port number to pass to the Provider used in connecting to the server.
--truststore-file|-t path

The file containing trusted certificates. Because this file usually itself has a password, you can specify that password as the special user TRUSTSTORE_PASSWORD within the password file. Alternately, you may be prompted for it if the --prompt option is used.

This option is used for TLS connections (if the Provider supports them). You may use the Java keytool program to create a truststore file.
--protocol|-r protocol-id The protocol-id selects the particular Provider that will be used to establish the connection.
--options|-o name=value[,name=value]* An arbitrarily long list of options to pass to the protocol Provider. The names and values of these options are protocol-specific. See the documentation for the Provider for details.

 

Notes:

  • if --host is not specified, then localhost is used.
  • if --protocol is not specified, the jmxmp is used.
  • if user and password-file are not specified, no user and password are supplied to the JMXConnectorProvider
  • additional options as name/value pairs may be specified and will be passed to the JMX connector as additional configuration data.

If specified, the --user option is used to lookup a password in the password file. The file specified by --password-file option should be of the following format:

user1=password1
user2=password2

Examples

1. Connect to a server using jmxmp at port 9377 with no username/password and give it the name 'test':

jmxcmd> connect --host=myhost --port=9377 --protocol=jmxmp test

2. Connect to a server using "funny" protocol and having several additional options:

jmxcmd> connect --host=myhost --port=9377 --protocol=funny --user=admin --password-file=test.pf --options=fastmode=true,secure=true,timeout=120 test

 

2.13 listen--listen for JMX Notifications

define-listener --name|-n name [--file|-f=filename] [--stdout|-t] [target[ target]*]
delete-listener name
start-listener name
stop-listener name
list-listeners

The listen command listens for notifications emitted by MBeans. This command is to be preferred over the monitor command when it is known that an MBean uses notifications. Information about an MBean's notifications can be obtained by using the inspect command.

Option Explanation
--name|-n Name of the listener eg "listener1"
--file | -f filename Emits all notifications to the specified file.
--stdout Emits all notifications to stdout. If --file is not specified, this need not be specified. However, if --file is specified, then this option can force dual output to both a file and to stdout.
Examples

1. Start listening to notifications on MBeans denoted by the alias MyMBeans:

jmxcmd> define-listener --name listener1 MyMBeans
jmxcmd> start-listening listener1

 

2.14 monitor--watch MBean attributes for changes

define-monitor --name|-n name [--attributes|-a=attr-expr[,attr-expr]*] [--interval|-i seconds] [--file|-f=filename] [--stdout|-t] target[ target]*
start-monitor name
stop-monitor name
list-monitors

The monitor command polls for changes to MBean Attributes This command can be used on any MBean, but is the only option for monitoring MBeans that do not support notifications. If an MBean supports notifications, the listen command is more efficient.

If no attributes are specified, then all attributes are monitored. Attributes may be specified that apply to only some of the target MBeans; only Attributes that actually exist are monitored for a particular MBean. The Attribute list is evaluated once only; if an MBean acquires new Attributes while it is being monitoring, they will not be included.

Option Explanation
--name | -n name Required option for specifying the name of the monitor being defined eg "monitor1".
--attributes | -a

A comma-separated list of Attribute names to be monitored. The following special values are available:

  • * = all Attributes
  • *r = all read-only Attributes
  • *w = all writeable Attributes
  • a "*" is interpreted as a wildcard character in any other situation

Note that use of "*" may require quoting from within a command-line shell (but not within jmxcmd shell).
--file | -f filename Emits all notifications to the specified file.
--stdout Emits all notifications to stdout. If --file is not specified, this need not be specified. However, if --file is specified, then this option can force dual output to both a file and to stdout.
--interval | -i The interval, in seconds between polling of the MBeans.
Examples

1. Start monitoring all Attributes on MBeans denoted by the alias MyMBeans:

jmxcmd> define-monitor --name m1 --attributes=* MyMBeans
jmxcmd> start-monitor m1

2. Start monitoring all read-only Attributes on MBeans denoted by the alias MyMBeans:

jmxcmd> define-monitor --name m2 --attributes=*r MyMBeans
jmxcmd> start-monitor m2

3. Start monitoring all writeable Attributes on MBeans denoted by the alias MyMBeans:

jmxcmd> define-monitor --name m3 --attributes=*w MyMBeans
jmxcmd> start-monitor m3

4. Start monitoring the Attribute "RequestCount" and "PoolSize" on MBeans denoted by the alias MyMBeans:

jmxcmd> define-monitor --name m4 -a=RequestCount,PoolSize MyMBeans
jmxcmd> start-monitor m4

5. Start monitoring all Attributes containing "Count" on MBeans denoted by the alias MyMBeans:

jmxcmd> define-monitor --name m5 -a=*Count* MyMBeans
jmxcmd> start-monitor m5

 

2.15 create-target-alias/delete-target-alias/list-target-aliases--manage persistent names for MBean(s)

create-target-alias [--replace|-R] [--resolve] name=value[ name=value]*
delete-target-alias alias-name[ alias-name]*
show-target-alias alias-name
resolve-target-alias alias-name

A target alias is a name for one or more values, each of which may be an ObjectName, and ObjectName pattern, or another target-alias. While no special interpretation is given to the value supplied when creating an alias, an alias must ultimately resolve to an ObjectName or ObjectName pattern. Aliases may contain other aliases in any combination with ObjectNames.

Once created, an alias may be used anywhere an ObjectName may be used. Aliases are persisted in the file system, typically in the user's home directory.

Options for create-target-alias:
Option Explanation
--replace | -k Replaces an existing alias. If not specified, it is an error to try to create an alias name that already exists.
--resolve | -r Interpret the value part of the alias immediately, resolving to fully-qualified ObjectNames. The resulting ObjectName(s) become the value(s) of the alias and are the same as what would be returned from resolve-alias.
Examples

1. Create an alias named "test" with value ":type=test,name=TestStandard"

jmxcmd> create-target-alias test=:type=test,name=TestStandard

2. Create an alias named "system" with value "system:*"

jmxcmd> create-target-alias system=system:*

3. Create an alias named "all-test" with value "*:type=test"

jmxcmd> create-target-alias all-test=*:type=test

4. Delete the alias "MyMBeans"

jmxcmd> delete-target-alias MyMBeans

5. Show the value of the alias MyMBeans, but without resolving the value to ObjectNames.

jmxcmd> show-target-alias MyMBeans

6. Show the ObjectNames to which the alias ultimately resolves.

jmxcmd> resolve-target-alias MyMBeans

2.19 mbs--manage in-process MBeanServers

start-mbs [--init|i=init-classname[,init-classname]*] [--mbeans|m=mbean-classname[,mbean-classname]*] server-name
stop-mbs server-name
list-mbs

load-jmxmp-connector [--port|-p port] [--keystore-file|-k path] [--acl-file|-a path] [--auth-file|-u path] [--password-file|-f path] [--prompt|-P] [object-name]

load-rmi-connector [--rmi-registry-port|-R port]

load-html-adapter [--port|-p port]

There are several variants here, described in the following table:

Name Explanation
start-mbs

Start an MBeanServer, optionally running an MBeanServerIniter and/or loading MBeans
stop-mbs Unregister all MBeans in the specified in-process MBeanServer, and remove it from the list
list-mbs List all in-process MBeanServers.
load-jmxmp-connector Loads the JSR160 jmxmp connector as an MBean into the current connection's MBeanServer and starts it. This operation requires that the current connection be to an in-process MBeanServer if security is to be used
load-rmi-connector Loads the JSR160 rmi connector as an MBean into the current connection's MBeanServer and starts it. This operation requires that the current connection be to an in-process MBeanServer.
load-html-adapter

Loads the JMX RI html adapter as an MBean into the current connection's MBeanServer and starts it. This operation requires that the current connection be to an in-process MBeanServer. After loading, a browser may be used to observe the MBeans using:

http://host:port

start-mbs

The initer(s) (if any) must implement MBeanServerIniter. The MBeans (if any) must have a no-argument constructor. They will be registered with a default name of ":server=server-name,classname=classname,id=nnn", where server-name is server-name as specified to mbs-start, classname is the name of the mbean's java class, and id is an arbitrary identitifier. Of course, the MBean may choose to override this name in preRegister().

Examples

1. Create an MBeanServer named "MyServer" and then connect to it:

jmxcmd> start-mbs MyServer
jmxcmd> connect MyServer

2. Same as (1), but run an initer and load 2 mbeans:

jmxcmd> start-mbs --init=com.myco.MyIniter --mbeans=com.myco.mbeans.MyMBean1,com.myco.mbeans.MyMBean2 MyServer

load-jmxmp-connector

Load the JSR 160 jmxmp Connector. The Connector may be loaded with no security by omitting all options (--port may still be used). To use without SSL/TLS, specify the --acl-file only. To use with TLS, you must include additional options.

Name Explanation
--port|-p port The port number to use
--keystore-file|-k path The file path to a Java KeyStore file, which must contain a key pair.
--acl-file|-a path The file path to an access-control-list file, consisting of pairs of role names and passwords.
--auth-file|-u path The file path to an authorization file, consisting of role names and valid operations.
--password-file|-f path The file path to a password file. An entry with the name "keystore-password" and "key-password" should be present so as to be able to unlock the keystore and the key within it. Alternately, the --prompt option may be used.
--prompt|-P Prompt for the keystore and key passwords instead of reading from a file.

2.21 create-proxies--create local proxy MBeans

create-proxies [--new-names|-n] --mbean-server|-m local-mbeanserver-nametarget [ target]*

Name Explanation
--mbean-server|-m Specify the name of the local MBeanServer which should contain the new proxy MBeans. You can create an MBeanServer using start-mbs.
--new-names | -n By default MBean proxy creation will fail if an MBean with the same name already exists in the target (destination) MBeanServer. If this option is specified, then each proxy ObjectName will contain an additional key to make it unique. This allows the proxying of MBeans with identical names from multiple servers (such as Sun's Java System Appserver) into the same MBeanServer.

You should first connect to the MBeanServer that contains the MBean(s) for which you want proxies. Then, if you don't already have an in-process MBeanServer, you should create one. Finally, issue the create-proxies command. See the example below.

Examples

1. Create a local MBeanServer "mbs1" and then create proxies for all MBeans found in the server "MyServer".

jmxcmd> connect MyServer
jmxcmd> start-mbs mbs1
jmxcmd> create-proxies --mbean-server mbs1 *

2. Commands detail

2.1 get --display one or more attributes on the specified target(s)

get attr[,attr]* [target]+

The 'get' command displays MBean attributes for all specified target(s). If no targets are specified, then the default target it used, as last set by the 'target' command. Attributes should be specified in a comma-separated list. In addition, the following special targets are supported:

Target Meaning
* All attributes
*r all read-only attributes
*w all writeable attributes

The get command is robust in the face of failures; failures in getting some Attributes of an MBean will not prevent the other Attributes from being accessed. Situations that can cause this include non-serializable Attributes or Attributes for cannot be deserialized due to the absence of client-side classes for them.

Examples

1. Get all attributes on all MBeans:

jmxcmd> get * *

Note that if you invoke jmxcmd from within a command shell, then you will need to quote the "*".

2. Get the 'Count' and 'Timeout' attributes on the MBean(s) referred to by the alias 'MyMBean':

jmxcmd> get Count,Timeout MyMBean

 

2.2 set --set one or more attributes on the specified target(s)

jmxcmd> set attr=value[,attr=value]* [target]+

The 'set' command sets the specified MBean attributes for all specified target(s). If no targets are specified, then the default target it used, as last set by the 'target' command. Attributes should be specified as a series of name-value pairs in a comma-separated list. In addition, the following special targets are supported:

Examples

1. Set the 'Count' and 'Timeout' attributes on the MBean(s) referred to by the alias 'MyMBean':

jmxcmd> set Timeout=10,Count=20 MyMBean

Note that the comma-separated list of name-value pairs must not include white space before or after the ','.

 

2.3 find --display MBeans matching name, pattern, or alias

find [--terse] [--wild-prop|-w=<expr=expr> | --jregex-prop|-j=<expr=expr>] [--current|-c ] [--add|-a] [--remove|-r] [target]*

find --current|-c [--terse]

Finds mbeans within the set of MBeans denoted by the target(s). Each operand to the find command may be a fully qualified ObjectName, an ObjectName pattern, or an alias. The set of these operands and ultimately what they resolve to defines the starting set upon which any further selection is done. Certain special characters may be used with the targets:

  • the '!' character may be used to reverse the set that the target selects.
  • with more than one operand, the '+', '-' and '&' characters modify the set of targets so far, from left to right. The '-' operator substracts the target from the set so far, the '+' operator adds to it (same as no qualifier), and the '&' character performs a logical and of the target and the set so far.

The MBeans found will be displayed unless the --terse option is used, in which case only a total count will be given.

The 'find' command automatically maintains the alias "last-found" with the results of its last invocation.

Option Explanation
--terse | -t display a count only of the MBeans found
--current | -c display the results of the last find (current set). This will be the same as the contents of the alias "last-found".
--add | -a add the results of this invocation to the current set
--remove | -r remove the results of this invocation from the current set
--wild-prop | -w utilize a wildcard expression for MBean Property name(s) or value(s). The only allowed wildcard is the '*' character.
--jregex-prop | -j utilize a regular expression for MBean Property name(s) or value(s) based on java.util.regex patterns
--operations | -o comma-separated list of (possibly wildcarded) expression(s) for operation names
--attributes | -a comma-separated list of (possibly wildcarded) expression(s) for Attribute names

When specifying --attributes or --operations, MBeans that match any of the operations or attributes are selected (logical OR). The same holds true for --regex and --jave-regex.

Examples

1. Display all MBeans:

jmxcmd> find

2. Display all MBean in the default domain:

jmxcmd> find :*

3. Display all MBeans whose ObjectName contains the property 'type' with value 'test'

jmxcmd> find type=test

4. Displays all MBeans whose ObjectName contains a property whose name starts with 'n' and whose value starts with 'test'

jmxcmd> find --wild-prop=n*=test* *:*

5. Displays all MBeans not denoted by the alias "MyMBeans":

jmxcmd> find !MyMBeans

6. Displays all MBeans which have the property-pair "category=config" and that are also in the domain "Test":

jmxcmd> find category=config &Test:*

7. Displays all MBeans that have an operation containing the string "Test", "test", or "TEST" or an attribute containing "Count" or "count" in the default domain that don't have an ObjectName property "category=config":

jmxcmd> find --operations=*Test*,*test*,*TEST* --attributes=*count*,*Count* :* -category=config

9. Inspects all MBeans last found with the find command

jmxcmd> inspect last-found

 

2.4 inspect --display MBeanInfo (attributes, operations, constructors, notifications)

inspect [--all|-A] [--no-description|-d] [--attributes|-a=attr[,attr]*] [--operations|-o=op[,op]* ] [--constructors|-c] [--notifications|-n=notif[,notif]*] [--include--nomatch|-i] [target]*

One or more targets may be specified. If no targets are specified, then the default target is used. All matching MBeans will be displayed with their fully-qualified names.

Option Explanation
--all | -A display all available information
--no-description | -d omit descriptions (avoids clutter if there are none)
--attributes | -a display the specified attributes (* = all attributes). A comma-separated list of attribute names may be used, and each of them may be wildcarded using the "*" character.

2.5 invoke --invoke an MBean operation

opname:[arg[,arg]*] [target[ target]*]
alias-name.opname:[arg[,arg]*]

opname([arg[,arg]*) [target[ target]*]
alias-name.opname([arg[,arg]*)

In this syntax, 'opname' is an operation name, alias-name is an alias as created by the create-alias command, 'arg' is an argument value or a named-argument (name/value pair) and 'target' is either an alias or ObjectName. An invocation uses wither Ordered Invocation, in which the order of the parameters is explicit according to the order they are listed, or Named Invocation, in which parameter order is determined by the MBeanParamerInfo using the parameter names. Note that Named invocation is not available if an MBean does not supply parameter names. You can use the inspect command to see if parameters have names.

Named invocation also works for operations taking a Properties object; excess parameters are supplied to the operation in the Properties object.
All forms of invocation use the MBeanInfo to determine the correct match for an operation.

For an operation to be available for invocation, it must restrict its use of data types to the following:

  • char, byte, short, int, long, float, double
  • Character, Byte, Short, Integer, Long, Float, Double, Number, BigNumber
  • String
  • Object
  • Properties
  • [] all arrays of the above types


Type-casts may be used, but are rarely needed. Use a type cast to force a number to a String or int to Integer, etc. Arrays are denoted using curly braces, and may be nested. Examples of arrays:

{1,2,3}
{hello,there}
{hello,1,there,2}
{{1,2},{3,4}}

If a type-cast is applied to an array, then all elements of that array must be compatible with it. The following type-cast
forces the value to be converted to an array of String (which would match 'String []' in an operation):

(String){1,2,3}


Strings may be quoted with the double-quote character '"'. This is not required, but can be useful to force a value's type
to be a String where it could otherwise be interpreted as a number.

Target(s) of the invocation may be specified in three ways:

  • implicitly by omission (the default target will be used as set by the target command);
  • explicitly as operands ;
  • explicitly as an alias name preceeding the invoked operation.

The operation may be specified using two syntactical forms:

  • the ':' form
  • the method invocation form ()

All forms are equivalent in function.

This table lists all variants:

# Variant Comments
1 opname:[arg[,arg]*] [target]+ invokes on one or more specified targets, which may be aliases or ObjectNames
2 opname:[arg[,arg]*] invokes on the default target(s) as set by the target command.
3 opname([arg[,arg]*]) [target]+ same as variant 1, but with () instead of ':'
4 opname(arg[,arg]*]) same as variant 2, but with () instead of ':'
5 alias-name.opname:[arg[,arg]*] invokes on the target(s) defined by the alias 'alias-name'.
6 alias-name.opname([arg[,arg]*]) Same as variant 5, but with () instead of ':'


Option Explanation
Examples

1. Invoke the start() operation an an object named "mydomain:name=controller":

jmxcmd> start() mydomain:name=controller

jmxcmd> start: mydomain:name=controller

2. Invoke the add( int x, int y) operation on an mbean with the alias 'math'

jmxcmd> add:3,4 math

jmxcmd> math.add(3,4)

2.12 connect--manage connections to MBeanServers

connect [--host|-h host] [--port|-p port] [--protocol|-r prot] [--user|-u username --password-file|-f filename] [--options|-o key=value[,key=value]*] [connection-name]
connect
connect connection-name
connect --list|-l
list-connections
close-connection [connection-name]

There are several variants here, described in the following table:

Name Explanation
connect Connect to the default server. The default server is the one last accessed.
connect connection-name Connect to a server, using the parameters with the name connection-name. The connection-name must have been previously created.
connect [<options>] Make a new connection using the specified parameters and optionally name it.
connect --list | -l
 List all connections configurations and open connections.
list-connections Same as "connect -l" above.
close-connection [connection-name] Close the named connection, or close the default one if no name is supplied

When jmxcmd makes a connection to an MBeanServer, it relies on pluggable modules implementing the JMXConnectorProvider interface (com.sun.cli.jmxcmd.spi.JMXConnectorProvider). Each connector has its own parameters, which may be a subset, superset, or alternative set to the ones shown in the above syntax. However, for user convenience when working with more than one connector, certain options are made available as common names. These include host, port, user, and password-file. The protocol option is always used to specify the protocol, but defaults to jmxmp, the TCP/IP based JSR 160 protocol for accessing remote MBeanServers.

Notes:

  • if --host is not specified, then localhost is used.
  • if --protocol is not specified, the jmxmp is used.
  • if user and password-file are not specified, no user and password are supplied to the JMXConnectorProvider
  • additional options as name/value pairs may be specified and will be passed to the JMX connector as additional configuration data.

If specified, the --user option is used to lookup a password in the password file. The file specified by --password-file option should be of the following format:

user1=password1
user2=password2

Examples

1. Connect to a server using jmxmp at port 9377 with no username/password and give it the name 'test':

jmxcmd> connect --host=myhost --port=9377 --protocol=jmxmp test

2. Connect to a server using "funny" protocol and having several additional options:

jmxcmd> connect --host=myhost --port=9377 --protocol=funny --user=admin --password-file=test.pf --options=fastmode=true,secure=true,timeout=120 test

 

2.13 listen--listen for JMX Notifications

The listen command listens for notifications emitted by MBeans. This command is to be preferred over the monitor command when it is known that an MBean uses notifications. Information about an MBean's notifications can be obtained by using the inspect command.

Option Explanation
--stop | -s Stops listening to the specified target(s).
--pause | -p Pauses all listening for all targets.
--file | -f filename Emits all notifications to the specified file.
--stdout Emits all notifications to stdout. If --file is not specified, this need not be specified. However, if --file is specified, then this option can force dual output to both a file and to stdout.

2.14 monitor--watch MBean attributes for changes

monitor [ --stop|-s | --pause|-p ] [--attributes=attr-name[,attr-name]*] [--file|-f=filename] [--stdout|-t] [--status|-u] target [[target]+]

The monitor command polls for changes to MBean Attributes This command can be used on any MBean, but is the only option for monitoring MBeans that do not support notifications. If an MBean supports notifications, the listen command is more efficient.

If no attributes are specified, then all attributes are monitored. Attributes may be specified that apply to only some of the target MBeans; only Attributes that actually exist are monitored for a particular MBean. The Attribute list is evaluated once only; if an MBean acquires new Attributes they will not be included.

Option Explanation
--attributes | -a

A comma-separated list of Attribute names to be monitored. The following special values are available:

  • * = all Attributes
  • *r = all read-only Attributes
  • *w = all writeable Attributes

Note that use of "*" may require quoting from within a command-line shell (but not within jmxcmd shell).
--file | -f filename Emits all notifications to the specified file.
--stdout Emits all notifications to stdout. If --file is not specified, this need not be specified. However, if --file is specified, then this option can force dual output to both a file and to stdout.
--status | -s Show what is being monitored. This option may only be used by itself.
--poll-interval | -i The interval, in seconds between polling of the MBeans.
--stop | -s Stops monitoring everything.
--pause | -p Pauses monitoring for everything.
--resume | -r Resumes monitoring for everything.
Examples

1. Start monitoring all Attributes on MBeans denoted by the alias MyMBeans:

jmxcmd> monitor --attributes=* MyMBeans

2. Start monitoring all read-only Attributes on MBeans denoted by the alias MyMBeans:

jmxcmd> monitor --attributes=*r MyMBeans

3. Start monitoring all writeable Attributes on MBeans denoted by the alias MyMBeans:

jmxcmd> monitor -a=*w MyMBeans

4. Start monitoring the Attribute "RequestCount" and "PoolSize" on MBeans denoted by the alias MyMBeans:

jmxcmd> monitor -a=RequestCount,PoolSize MyMBeans

 

2.15 create-alias--create a persistent name for an alias, MBean name or pattern

create-alias [--replace|-R] name=value[ name=value]*

Creates aliases--one for each name/value pair. No interpretation is given to the value supplied when creating an alias. However, an alias must ultimately resolve to an ObjectName or ObjectName pattern. Aliases may contain other aliases in any combination with ObjectNames. The only restriction is that the space character is reserved as a delimiter. Therefore, ObjectNames containing spaces may not be aliased.

Once created, an alias may be used anywhere an ObjectName may be used. Aliases are persisted in the file system, typically in the user's home directory.

Option Explanation
--replace | -R Replaces an existing alias. If not specified, it is an error to try to use an alias name that already exists.
Examples

1. Create an alias named "test" with value ":type=test,name=TestStandard"

jmxcmd> create-alias test=:type=test,name=TestStandard

2. Create an alias named "system" with value "system:*"

jmxcmd> create-alias system=system:*

2. Create an alias named "all-test" with value "*:type=test"

jmxcmd> create-alias all-test=*:type=test

2.21 create-proxies--create local proxy MBeans

create-proxies [--new-names|-n] --mbean-server|-m local-mbeanserver-nametarget [ target]*

Name Explanation
--mbean-server|-m Specify the name of the local MBeanServer which should contain the new proxy MBeans. You can create an MBeanServer using start-mbs.
--new-names | -n By default MBean proxy creation will fail if an MBean with the same name already exists in the target (destination) MBeanServer.
